# 问题修复后文档更新规范

> 适用场景：每次 Claude Code 分析并修复问题后，必须按本文件的步骤更新相关文档。
> 
> **执行时机**：问题修复完成、功能验证通过后，立即执行以下步骤，不得跳过。

---

## 执行步骤

### 第一步：判断问题根源类型

从以下类型中选择一个或多个：

| 类型 | 说明 | 示例 |
|------|------|------|
| 技术方案疏漏 | 技术方案文档中没有提到该配置或设计 | CORS 未配置、接口参数未定义 |
| 开发计划疏漏 | 开发计划中该任务描述不完整或缺少步骤 | 初始化任务未包含 CORS 配置步骤 |
| 环境适配问题 | Windows / 云数据库 / 特定版本的兼容性问题 | EPERM 文件锁、shadow database 权限 |
| 前后端集成问题 | 单独验证通过但联调时才暴露的问题 | 查询参数不一致、字段命名不匹配 |
| 依赖版本问题 | npm 包版本升级导致 API 或配置方式变化 | Prisma 7 配置方式与 Prisma 6 不兼容 |

---

### 第二步：更新技术方案文档

**文件路径**：`docs/hr-saas-technical-spec.md`

**触发条件**：问题类型为"技术方案疏漏"或"前后端集成问题"时必须更新。

**更新要求**：
- 找到最相关的章节，在该章节补充缺失的内容
- 如果没有合适的章节，在文档末尾新增章节
- 更新内容要具体，包含配置示例或代码片段
- 不得删除已有内容，只做补充

---

### 第三步：更新开发计划文档

**文件路径**：`docs/hr-saas-dev-plan.md`

**触发条件**：问题类型为"开发计划疏漏"或"环境适配问题"时必须更新。

**更新要求**：
- 如果是某个任务缺少步骤，在该任务的开发内容中补充
- 如果是通用的执行规范，在文档顶部"执行规则"部分新增一条
- 如果是验证方式有问题，更新对应任务的验证清单
- 规则编号按已有规则顺序递增

---

### 第四步：更新问题规避清单

**文件路径**：`docs/project-startup-checklist.md`

**必须执行，无论问题类型是什么。**

**4.1 在第六章问题记录表新增一行**

按以下格式填写：

| 问题 | 原因 | 解决方案 | 应在哪个环节预防 |
|------|------|---------|----------------|
| [问题简短描述] | [根本原因一句话] | [解决方案一句话] | [技术方案/开发计划/环境规划] |

**4.2 判断是否为通用问题**

如果该问题在其他项目（使用相同技术栈）中也可能出现，则在清单第一到五章对应分类中新增检查项：
- 检查项以 `- [ ]` 格式书写
- 注明【技术方案】【开发计划】或【两者都要】
- 描述要具体，说明怎么做，不能只说"注意xxx"

---

### 第五步：输出变更摘要

更新完成后，输出以下格式的摘要：

```
📋 文档更新摘要

问题类型：[类型]

更新内容：
- [文件名]：[修改了哪个章节/任务，补充了什么内容]
- [文件名]：[修改了哪个章节/任务，补充了什么内容]

是否为通用问题：[是/否]
[如果是，说明补充到了清单的哪个分类]
```

---

## 注意事项

1. **不要等到所有问题都修复后再统一更新文档**，每修复一个问题立即更新一次
2. **文档更新和代码修复同等重要**，不得以"先修复问题再说"为由跳过文档更新
3. **更新内容要对下一个执行这份文档的人有意义**，避免写"修复了某某问题"这种无意义的描述，要写清楚"应该怎么做"
4. **如果不确定归属哪个章节**，优先更新 `project-startup-checklist.md` 的第六章问题记录表，至少保证问题被记录下来
